面向对象

面向对象三要素

  • 继承：子类继承父类
  • 封装：数据的权限和保密
    • public：完全开放
    • protected：对子类和自己开放
    • private：对自己开放
  • 多态
    • 同一接口，不同表现，例如：父类有一个方法，它的子类们重写了该方法实现了不同的功能
    • 面向接口编程
    • JS 应用极少
    • 需要结合 JAVA 等语言的接口、重写、重载等功能

UML 类图

  • Unified Modeling Language 统一建模语言
  • 类图：UML 有很多图，这里指的是类图
    • 属性
    • 方法
  • 关系
    • 泛化：类之间的继承
    • 关联：类之间的引用

何为设计

  • 伴随着需求增加，设计的作用才能体现出来
  • 《UNIX/LINUX 设计哲学》
    • 十大准则
      • 小即是美
      • 让每个程序只做好一件事（大程序拆分成一个个小程序）
      • 快速建立原型（最基本的功能）
      • 舍弃搞效率而取可移植性（代码通用，可复制。复用性第一，性能第二。性能也可以通过硬件升级而提高）
      • 采用纯文本来存储数据（可读性）
      • 充分利用软件的杠杆效应（软件复用）
      • 使用 shell 脚本来提高杠杆效应和可移植性
      • 避免强制性的用户界面（用户界面是作为一个软件而存在的，它不应该存为一个系统的必备项。它的缺点如，占用内存。它可以作为一个插件插入，但不要强制与系统绑定）
      • 让每个程序都称为过滤器（也就是上面小而美、每个程序做好一件事的意思，就是把数据先到 A 进行处理，之后到 B 进行处理，再到 C。）
    • 其他小准则
      • 允许用户定制环境
      • 尽量是操作系统内核小而轻量化
      • 使用小写字母并尽量简短
      • 沉默是金（输入的数据类型不匹配，则输出 null 或空。反例就是，要求输入一个数字，但是没有输入数字会返回”这不是一个数字”）
      • 各部分之和大于整体（以小的个体组成一个大的整体。作用是，某个部分改变只改该部分，而不是改整个被高度耦合的整体）
      • 寻求 90% 的解决方案（一个产品不可能满足 100% 的用户需求，我只需要满足 90% 就行了。二八定律，花 20% 的成本解决 80% 的需求，而剩下 20% 的需求就需要花费大量的精力取解决了。）

一个例子：
沉默是金 + 让每个程序成为过滤器
指令 ls，grep 等就是一个个过滤器，过滤出需要的文件，如果没有匹配的文件控制台就没有输出。

SOLID 五大设计原则

  • S （single）- 单一职责原则
    • 一个程序只做一件事
    • 功能过于复杂就把它拆分开来
  • O （open） - 开发封闭原则
    • 对扩展开发，对修改封闭
    • 增加需求时，扩展新代码，而非修改已有的代码
    • 抽象得很合理，这是软件设计的终极目标
  • L （Liskov） - 李氏置换原则
    • 子类能覆盖父类
    • 所有能用父类的地方，也能用子类代替
    • JS 中使用较少（弱类型 & 继承使用较少）
  • I （interface）- 接口独立原则
    • 保持接口的单一独立，避免一个接口做了很多功能
    • J- S 中没有接口（typescript 例外），使用较少
    • 类似于单一职责原则，这里关注接口
  • D （dependence）- 依赖倒置原则
    • 面向接口编程，依赖于抽象而不依赖于具体（合作的时候只关注 UML 类图，内部的实现不用管，你爱怎么实现怎么实现）
    • 使用方只关注接口，而不是关注具体类的实现
    • JS 中使用较少（没有接口 & 弱类型）

从设计到模式
