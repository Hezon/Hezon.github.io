前端模块化发展

发表于 2019-08-29 | 分类于 前端模块化

在前端发展初期，对模块化的意识不强，顶多就是用自执行函数（IIFE）的形式减少变量命名冲突问题，但是会有文件的依赖不明确等问题。

后来，出现了 AMD 规范。

AMD

RequireJS 就是遵循 AMD 规范的一个模块化库。

它有两个核心的方法，define 和 require。用 define 定义模块，用 require 引用模块。

它的优势是可以减少全局变量污染和明确 JS 文件的依赖关系，并且 JS 文件异步加载。

同时期出现了 CMD，它和 AMD 类似，对应的库是 SeaJS。

CommonJS

后来，由于 nodejs 的出现而有了 CommonJS 规范。它在 nodejs 和 webpack 中被大量使用。

它的使用方法是，用 module.exports 暴露模块，用 require 引用模块。

它不像 ReqireJS 那样是异步加载文件，CommonJS 是同步加载文件的。因此，CommonJS 直接在页面中使用并不好，会阻塞浏览器渲染。

UMD

UMD 是一种跨平台的模块化方案，兼容 AMD、CommonJS 等模块化语法。

ES6 Modules

ES6 模块化语法一般搭配 webpack 构建工具使用，再结合 babel 做兼容处理。

它的使用方式是，export 导出模块，import 引入模块。

还有一种模块打包工具是 rollup，它比 webapck 轻便许多，打包出来的代码也比 webpack 少一些。vue 和 react 也使用了 rollup 进行打包。

总结

前端模块化的发展，途中经过了 IIFE、AMD、CMD、CommonJS、UMD，最后到了 ES6 Modules。模块化的优势是明确了的各个文件的依赖关系，并且减少了变量冲突。

函数式编程

介绍

函数式编程是编程范式的其中一种。编程范式分为两类，命令式编程和声明式编程，而函数式编程属于后者。命令式编程注重过程，而声明式编程注重结果。

函数式编程的优势在于可以更好的模块化，而模块化有利于开发维护、代码复用、单元测试等。

下面开始讲函数式编程的四个模块：纯函数、高阶函数、柯里化函数和代码组合。

介绍

  • 不修改外部状态
  • 函数的输出依赖于输入的参数，与外部状态无关

JavaScript 设计模式 - 适配器模式

发表于 2019-08-07 | 分类于 设计模式

介绍

适配器模式用于适配两个不兼容的接口。比如 USB 数据线就是一个适配器，手机通过这个适配器就可以电脑连接，进行数据通信。适配器模式差不多就是这个意思。

场景

适配器模式通常用于兼容旧接口。当我们在接手一些旧项目的时候，自己封装了一个新的方法去替代旧的方法，但是这两个方法的使用方式不一样，这种情况就可以用适配器去解决。

举个例子，在旧项目中，我们写了一个新的 ajax 方法去替代旧的方法，两种方法传参形式都相同，但是旧的方法是以 $.ajax() 形式去调用的。我们不推荐把项目中的每个 $.ajax() 都改为 ajax()，所以解决方案就是使用适配器模式。

何为设计

  • 伴随着需求增加，设计的作用才能体现出来
  • 《UNIX/LINUX 设计哲学》
    • 十大准则
      • 小即是美
      • 让每个程序只做好一件事（大程序拆分成一个个小程序）
      • 快速建立原型（最基本的功能）
      • 舍弃搞效率而取可移植性（代码通用，可复制。复用性第一，性能第二。性能也可以通过硬件升级而提高）
      • 采用纯文本来存储数据（可读性）
      • 充分利用软件的杠杆效应（软件复用）
      • 使用 shell 脚本来提高杠杆效应和可移植性
      • 避免强制性的用户界面（用户界面是作为一个软件而存在的，它不应该存为一个系统的必备项。它的缺点如，占用内存。它可以作为一个插件插入，但不要强制与系统绑定）
      • 让每个程序都称为过滤器（也就是上面小而美、每个程序做好一件事的意思，就是把数据先到 A 进行处理，之后到 B 进行处理，再到 C。）
    • 其他小准则
      • 允许用户定制环境
      • 尽量是操作系统内核小而轻量化
      • 使用小写字母并尽量简短
      • 沉默是金（输入的数据类型不匹配，则输出 null 或空。反例就是，要求输入一个数字，但是没有输入数字会返回”这不是一个数字”）
      • 各部分之和大于整体（以小的个体组成一个大的整体。作用是，某个部分改变只改该部分，而不是改整个被高度耦合的整体）
      • 寻求 90% 的解决方案（一个产品不可能满足 100% 的用户需求，我只需要满足 90% 就行了。二八定律，花 20% 的成本解决 80% 的需求，而剩下 20% 的需求就需要花费大量的精力取解决了。）

SOLID 五大设计原则

  • S （single）- 单一职责原则
    • 一个程序只做一件事
    • 功能过于复杂就把它拆分开来
  • O （open） - 开发封闭原则
    • 对扩展开发，对修改封闭
    • 增加需求时，扩展新代码，而非修改已有的代码
    • 抽象得很合理，这是软件设计的终极目标
  • L （Liskov） - 李氏置换原则
    • 子类能覆盖父类
    • 所有能用父类的地方，也能用子类代替
    • JS 中使用较少（弱类型 & 继承使用较少）
  • I （interface）- 接口独立原则
    • 保持接口的单一独立，避免一个接口做了很多功能
    • J- S 中没有接口（typescript 例外），使用较少
    • 类似于单一职责原则，这里关注接口
  • D （dependence）- 依赖倒置原则
    • 面向接口编程，依赖于抽象而不依赖于具体（合作的时候只关注 UML 类图，内部的实现不用管，你爱怎么实现怎么实现）
    • 使用方只关注接口，而不是关注具体类的实现
    • JS 中使用较少（没有接口 & 弱类型）

JS 引擎线程负责解析运行 javascript 脚本，在执行代码的过程中若调用一些浏览器 api，就把这些调用浏览器 api 的任务分发给其他相关的线程去运行。运行结束后，这些任务回调函数就会进入任务队列等待。在 js 执行栈为空后，这些任务就通过事件循环进入执行栈运行。

定时器触发线程

当 js 主线程执行栈执行了 setTimeout 或 setInterval 时，就会把这个定时任务交给定时器触发线程去执行。计时结束后，把回调函数添加到任务队列里。

事件触发线程

事件触发线程用来监听一些浏览器的操作，比如鼠标事件、键盘事件。当事件触发后，就把回调函数放入任务队列中。

HTTP 异步请求线程

该线程负责 http 请求，在得到相应时，把回调函数放入任务队列中，等待 js 引擎获取执行。

CORS 头信息总结

发表于 2019-07-07 | 分类于 技术周报

简单请求

若客户端发送的 http 请求的头信息不超过以下几种字段，且 http 方法为 get、head、post 之一，则浏览器判断该请求为简单请求：

  1. Accept：客户端希望接收的MIME类型（type/subtype），如：text/plain。
  2. Accept-Language：客户端希望接收的语言。
  3. Content-Language：访问者希望采用的语言。
  4. Content-Type：只限于三个值 application/x-www-form-urlencoded、multipart/form-data、text/plain。

请求头信息

Origin

  • 表示本次请求来自哪个源。
  • 若浏览器识别这是一次简单请求，它会自动在请求头中添加 origin 字段。

响应头信息

Accept-control-Allow-Origin

  • 必要字段。
  • 表示服务器允许接收哪些源的请求。

Accept-control-Allow-credentials

  • 可选字段。
  • 表示是否需要发送 cookie，为布尔值。
    当该字段设为 true 时，Access-Control-Allow-Origin 不能设为星号，必须指定明确的、与请求网页一致的域名。并且，客户端在发送请求时需要配置：xhr.withCredentials = true

Accept-control-Expose-Header

  • 可选字段。
  • 表示服务器允许客户端访问的自定义头信息。
  • XMLHttpRequest 对象的 getResponseHeader() 方法只能拿到6个基本字段：Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragma。客户端想要访问其他头信息，就需要服务端设置 Accept-control-Expose-Header 。

非简单请求

OPTIONS 预检请求头信息

Origin

  • 同简单请求。

Access-Control-Request-Method

  • 必要字段。
  • 表示客户端会用到的 HTTP 方法。

Access-Control-Request-Headers

  • 可选字段。
  • 表示在预检请求之后，客户端发送实际请求时想要携带的头信息字段。服务端在响应时，用字段 Access-Control-Allow-Headers 来表示所允许的头信息字段。

预检请求响应头信息

Access-Control-Allow-Origin

  • 同简单请求。

Access-Control-Allow-Methods

  • 必要字段。
  • 表示服务端所支持的方法。

Access-Control-Allow-Headers

  • 若预检请求携带 Access-Control-Request-Headers 字段，则必须。
  • 表示服务端允许接收的头信息字段。

Access-Control-Allow-Credentials

  • 同简单请求。

Access-Control-Max-Age

  • 可选字段。
  • 表示本次预检请求的有效期，单位为秒。

在预检请求之后，发送的实际请求，客户端每次都携带 Origin 字段，而服务端每次都响应 Access-Control-Allow-Origin 字段。
