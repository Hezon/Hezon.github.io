面向对象

面向对象三要素

  • 继承：子类继承父类
  • 封装：数据的权限和保密
    • public：完全开放
    • protected：对子类和自己开放
    • private：对自己开放
  • 多态
    • 同一接口，不同表现，例如：父类有一个方法，它的子类们重写了该方法实现了不同的功能
    • 面向接口编程
    • JS 应用极少
    • 需要结合 JAVA 等语言的接口、重写、重载等功能

何为设计

  • 伴随着需求增加，设计的作用才能体现出来
  • 《UNIX/LINUX 设计哲学》
    • 十大准则
      • 小即是美
      • 让每个程序只做好一件事（大程序拆分成一个个小程序）
      • 快速建立原型（最基本的功能）
      • 舍弃搞效率而取可移植性（代码通用，可复制。复用性第一，性能第二。性能也可以通过硬件升级而提高）
      • 采用纯文本来存储数据（可读性）
      • 充分利用软件的杠杆效应（软件复用）
      • 使用 shell 脚本来提高杠杆效应和可移植性
      • 避免强制性的用户界面（用户界面是作为一个软件而存在的，它不应该存为一个系统的必备项。它的缺点如，占用内存。它可以作为一个插件插入，但不要强制与系统绑定）
      • 让每个程序都称为过滤器（也就是上面小而美、每个程序做好一件事的意思，就是把数据先到 A 进行处理，之后到 B 进行处理，再到 C。）
    • 其他小准则
      • 允许用户定制环境
      • 尽量是操作系统内核小而轻量化
      • 使用小写字母并尽量简短
      • 沉默是金（输入的数据类型不匹配，则输出 null 或空。反例就是，要求输入一个数字，但是没有输入数字会返回”这不是一个数字”）
      • 各部分之和大于整体（以小的个体组成一个大的整体。作用是，某个部分改变只改该部分，而不是改整个被高度耦合的整体）
      • 寻求 90% 的解决方案（一个产品不可能满足 100% 的用户需求，我只需要满足 90% 就行了。二八定律，花 20% 的成本解决 80% 的需求，而剩下 20% 的需求就需要花费大量的精力取解决了。）

指令 ls，grep 等就是一个个过滤器，过滤出需要的文件，如果没有匹配的文件控制台就没有输出。

SOLID 五大设计原则

  • S （single）- 单一职责原则
    • 一个程序只做一件事
    • 功能过于复杂就把它拆分开来
  • O （open） - 开发封闭原则
    • 对扩展开发，对修改封闭
    • 增加需求时，扩展新代码，而非修改已有的代码
    • 抽象得很合理，这是软件设计的终极目标
  • L （Liskov） - 李氏置换原则
    • 子类能覆盖父类
    • 所有能用父类的地方，也能用子类代替
    • JS 中使用较少（弱类型 & 继承使用较少）
  • I （interface）- 接口独立原则
    • 保持接口的单一独立，避免一个接口做了很多功能
    • J- S 中没有接口（typescript 例外），使用较少
    • 类似于单一职责原则，这里关注接口
  • D （dependence）- 依赖倒置原则
    • 面向接口编程，依赖于抽象而不依赖于具体（合作的时候只关注 UML 类图，内部的实现不用管，你爱怎么实现怎么实现）
    • 使用方只关注接口，而不是关注具体类的实现
    • JS 中使用较少（没有接口 & 弱类型）

浏览器常驻线程分析

发表于 2019-07-14 | 分类于 技术周报

在浏览器内核里，有一些常驻线程，如下

  • GUI 渲染线程
  • JS 引擎线程
  • 定时器触发线程
  • 事件触发线程
  • http 异步请求线程

GUI 渲染线程

当页面需要重绘时，该线程就会被执行。

GUI 渲染线程和 js 引擎线程是互斥的。当 js 引擎线程在执行代码的时候，GUI 渲染线程被冻结。可以用代码测试一下：

JS 引擎线程

JS 引擎线程负责解析运行 javascript 脚本，在执行代码的过程中若调用一些浏览器 api，就把这些调用浏览器 api 的任务分发给其他相关的线程去运行。运行结束后，这些任务回调函数就会进入任务队列等待。在 js 执行栈为空后，这些任务就通过事件循环进入执行栈运行。

定时器触发线程

当 js 主线程执行栈执行了 setTimeout 或 setInterval 时，就会把这个定时任务交给定时器触发线程去执行。计时结束后，把回调函数添加到任务队列里。

事件触发线程

事件触发线程用来监听一些浏览器的操作，比如鼠标事件、键盘事件。当事件触发后，就把回调函数放入任务队列中。

HTTP 异步请求线程

该线程负责 http 请求，在得到相应时，把回调函数放入任务队列中，等待 js 引擎获取执行。

CORS 头信息总结

发表于 2019-07-07 | 分类于 技术周报

简单请求

若客户端发送的 http 请求的头信息不超过以下几种字段，且 http 方法为 get、head、post 之一，则浏览器判断该请求为简单请求：

  1. Accept：客户端希望接收的MIME类型（type/subtype），如：text/plain。
  2. Accept-Language：客户端希望接收的语言。
  3. Content-Language：访问者希望采用的语言。
  4. Content-Type：只限于三个值 application/x-www-form-urlencoded、multipart/form-data、text/plain。

请求头信息

Origin

  • 表示本次请求来自哪个源。
  • 若浏览器识别这是一次简单请求，它会自动在请求头中添加 origin 字段。

响应头信息

Accept-control-Allow-Origin

  • 必要字段。
  • 表示服务器允许接收哪些源的请求。

Accept-control-Allow-credentials

  • 可选字段。
  • 表示是否需要发送 cookie，为布尔值。
    当该字段设为 true 时，Access-Control-Allow-Origin 不能设为星号，必须指定明确的、与请求网页一致的域名。并且，客户端在发送请求时需要配置：xhr.withCredentials = true

Accept-control-Expose-Header

  • 可选字段。
  • 表示服务器允许客户端访问的自定义头信息。
  • XMLHttpRequest 对象的 getResponseHeader() 方法只能拿到6个基本字段：Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragma。客户端想要访问其他头信息，就需要服务端设置 Accept-control-Expose-Header 。

非简单请求

OPTIONS 预检请求头信息

Origin

  • 同简单请求。

Access-Control-Request-Method

  • 必要字段。
  • 表示客户端会用到的 HTTP 方法。

Access-Control-Request-Headers

  • 可选字段。
  • 表示在预检请求之后，客户端发送实际请求时想要携带的头信息字段。服务端在响应时，用字段 Access-Control-Allow-Headers 来表示所允许的头信息字段。

预检请求响应头信息

Access-Control-Allow-Origin

  • 同简单请求。

Access-Control-Allow-Methods

  • 必要字段。
  • 表示服务端所支持的方法。

Access-Control-Allow-Headers

  • 若预检请求携带 Access-Control-Request-Headers 字段，则必须。
  • 表示服务端允许接收的头信息字段。

Access-Control-Allow-Credentials

  • 同简单请求。

Access-Control-Max-Age

  • 可选字段。
  • 表示本次预检请求的有效期，单位为秒。

在预检请求之后，发送的实际请求，客户端每次都携带 Origin 字段，而服务端每次都响应 Access-Control-Allow-Origin 字段。

数据结构

栈

stack 具有两项基本操作：

  • push，添加元素到栈的末尾（顶端）
  • pop，移除栈末尾（顶端）的元素

以上的两个操作可以概况为：后进先出，LIFO（last in, first out）。

另外，还有一些其他操作，如：

  • peek，获取末尾元素
  • isFull, 是否满栈
  • isEmpty，是否为空栈
  • size，栈的尺寸
  • clear，清空栈

使用数组实现一个栈类

队列

queue 具有两项基本操作：

  • enqueue，添加元素到队列的后端
  • dequeue，移除队列前端的元素

以上的两个操作可以概况为：先进先出，FIFO（first in, first out）。

另外，还有一些其他操作，如：

  • front，获取前端的元素
  • isEmpty，是否为空队列
  • size，队列的长度
  • clear，清空队列

使用数组实现一个队列类

链表

链表是数据元素的线性集合，每个元素由一个存储元素本身的节点和一个指向下一个元素的引用组成。

相对数组，链表添加或移除元素的时候不需要移动其他元素，能更有效地从序列中的任何位置插入或删除元素。

数组的好处是可以直接访问任何位置的任何元素，而要想访问链表中间的一个元素，需要从起点开始迭代列表直到找到所需的元素。

web技术演化

发表于 2019-06-29 | 分类于 技术周报

在web诞生之初，它只是一个纯静态页面，只能提供展示作用。

之后，CGI出现，让网页有了动态元素。

再之后，PHP、ASP、JSP等技术的出现，它们更好的组织了web内容。

后来，MVC架构的概念被引入到web开发中来。模型Model用于封装与业务逻辑相关的数据和数据处理方法，视图View是数据的HTML展现，控制器Controller负责响应请求，协调Model和View。

接下来就是SPA时代。AJAX的出现使得页面可以进行局部刷新，而不是每次请求新的页面。

但是浏览器对AJAX的支持存在兼容问题。为了解决浏览器的各种兼容问题， jQuery 横空出世。

随着需求越来越复杂，系统功能越来越多，代码文件也越来越多，相互之间的依赖调用关系变得异常复杂。这时候有两个问题变得十分突出，js命名冲突问题和资源加载问题。这时出现了seajs、requirejs等模块化技术。

当一个系统的模块越来越多之后，模块内部的代码维护便是个大问题，为了让模块代码变得清晰，更易维护，相互之间没有紧密的耦合关系，MVC被应用到了前端。比如backbone.js。

后来前端架构继续演化，出现了MVVM模式，实现了双向数据绑定。开发者可以从的dom操作中解放出来。代表技术如 Vue。

传统的MV*框架存在着数据流不清晰的问题，因为View和Model直接的通信往往都是双向的。因此为了让数据流更加清楚，Facebook提出了单向数据流的Flux架构。

排序算法

快速排序

1.选择一个基数，如第一位数，和其他数对比，小的在左，相等的在中间（包括自己），大的在右。
2.把第一步左侧和右侧的数进行递归，左中右的数组拼接在一块。

选择排序

1.从第一位开始，对比出最大的一位数，取出它，放入第一位。
2.从第二位开始，对比出最大的一位数，取出它，放入第一位。
3.以此类推，直到完成排序。

冒泡排序

1.从左到右，对比相邻的两个数，调换位置，小的在左，大的在右。
2.重复第一步，但排除和最后一位对比。
3.重复第一步，排除和最后二位对比，以此类推。

插入排序

1.有两个集合，有序集合，和无序集合。
2.取无效集合的第一个数，与有序集合里的数对比，插入有序集合对应的位置。
3.重复第二步，直到无序集合的数全部插入有序集合。
